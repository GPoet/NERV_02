3 Architectural Approaches

Same tech stack (Tauri + React), different structural philosophy

A

Single Tauri App — Clean Slate

One self-contained Tauri app. Rust backend owns: WebSocket to OpenClaw, Claude CLI process spawning, MCP server registry, file system. React frontend with shadcn/ui. Connects to NERV API for skill dispatch (no duplication).

Pros

  • Simplest to build and ship
  • Full control, no coordination overhead
  • Fastest path to v1

Cons

  • Dashboard logic duplicated from NERV web app
  • Two separate codebases to maintain
B

Turborepo Monorepo — Shared Components ⭐ Recommended

Monorepo with packages/ui (shared shadcn components), packages/nerv-core (shared API client, types), apps/desktop (Tauri), apps/web (existing NERV dashboard). Desktop and web share the same component library and API hooks.

Pros

  • Build once, works in both apps
  • NERV web dashboard gets the new UI too
  • Clean path to eventually replace the web dashboard

Cons

  • More setup upfront (~1 extra day)
  • Tauri + Next.js in same repo needs care
C

Plugin Architecture — Extensible Panels

Core shell app with a panel registry. Each panel (MCP Inspector, OpenClaw Monitor, Session Viewer, Skill Dispatch) is an independent module that registers itself. New panels can be added without touching core.

Pros

  • Very extensible — add panels like skills
  • Clean separation of concerns

Cons

  • Over-engineered for current scope
  • Plugin API adds complexity before any value
  • Risk of building framework instead of app

My recommendation: B — The monorepo pays off immediately because you're already building the NERV dashboard. Shared components mean the MCP inspector and OpenClaw panel you build for desktop automatically work in the browser too. Option A is valid if you want to ship faster and don't mind two codebases. Avoid C for now.
